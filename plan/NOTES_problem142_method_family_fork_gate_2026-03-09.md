## Exhausted Current Family

For the purposes of this repository, the current exhausted family is the combined program built
from the following layers:

1. split upper/lower profile extraction from the currently audited sources;
2. matched-profile or dominance-style coupling from split data to a full asymptotic statement;
3. for `k = 3`, the modern almost-periodicity -> structured Bohr set -> density increment
   architecture, together with the same integer-case transfer losses and the same recurrence body.

So the following do **not** count as a new method family:

- better constants or better bookkeeping inside the same recurrence;
- localization or endgame reformulation inside the same architecture;
- new wrappers, new checker surfaces, or new gap packaging;
- a purported alternate transfer that leaves the decisive Bohr/rank loss unchanged.

## Admissible New Family Standard

A new method family must change at least one decisive layer above in an exponent-relevant way.

Admissible examples include:

- a direct asymptotic/counting route that bypasses the current split-to-matched-profile coupling;
- a higher-order uniformity / structured-decomposition route whose key theorem target is not the
  current almost-periodicity-to-Bohr-set bootstrap;
- a genuinely different ambient transfer or structured-object route that changes the present
  integer-case loss profile rather than merely renaming it.

That means:

- separate plan chain;
- no edits to the current theorem/checker/README status surfaces merely for scouting;
- no promotion of scouting targets into `Problem142Gap.lean` or public route status;
- immediate closure if the branch collapses back to the current exhausted family.
